# Reasoning Principles — expanded

These are what separate this skill from generic office-politics advice. The SKILL.md has the one-line versions; this file is the worked reasoning behind each, with synthetic illustrations.

---

### 1. Behavior ≠ motive

A person's *actions* are evidence; their *reasons* are inference. The whole artifact is built on this line. When you write "She scheduled the review without you," that's `[Observed]`. When you write "because she wants you sidelined," that's `[Inferred]` — and it's the part most likely to be wrong. Hold the line in the analysis too: a counterfactual that assumes a motive is only as good as that assumption. Make the assumption visible so it can be tested.

### 2. Steelman the antagonist

For every actor the user dislikes, surface the **competing benign read** and what would distinguish the two. The reason isn't fairness for its own sake — it's that the two readings imply *different moves*.

> *Synthetic:* The user says their manager "undercuts me in every leadership meeting." Malicious read: the manager is building a case to push the user out. Benign read: the manager does this to *everyone*, because looking decisive upward is how they self-soothe in front of their own boss. If it's the first, the counterfactual is "protect yourself / find an exit." If it's the second, the counterfactual is "give them a way to look decisive that includes you." Same behavior, opposite strategy. So: which is it, and what evidence would tell you?

Surface this as a fork to resolve, never as "you're wrong about them."

### 3. Reflexive epistemic humility

Turn the user's own scrutiny back on the user. The most common failure is hardening a confident read into a fact.

> *Synthetic:* "He's so sure of himself, which proves he doesn't actually understand the system." That inference leans on a folk version of the Dunning–Kruger effect — which is **contested** in the literature (some argue the classic pattern is partly a statistical artifact). So "his confidence proves his incompetence" is a *prior*, not a finding. Worth holding loosely.

Also catch the subtler ego games: e.g. a user who prides themselves on "I start from *I don't know*" may be running a quiet superiority move under the humble framing. Name it kindly; it usually lands because it's true.

### 4. Find seams, not villains

The most useful output is almost always structural, not a verdict on a person. A "villain" read is a dead end (you can't change them); a *seam* is actionable (you can reposition relative to it).

> *Synthetic:* Instead of "my skip is the problem," the seam is: "your performance rating is set by a chain you have no read on, while your day-to-day relevance depends on a team you don't report to. You're load-bearing between two systems that don't talk to each other." Now there's something to *do*.

### 5. Read revealed preference; don't resolve fuzzy targets abstractly

When the stated goal is mixed or self-contradictory, don't force a choice in the abstract — people are bad at introspecting on what they want. Instead, run branches for the different targets and let the user notice which *outcome* they react to.

> *Synthetic:* "I genuinely don't care about titles… but it'd be nice to finally get the senior bump." Don't ask them to rank those. Show the branch where they optimize for autonomy and the branch where they optimize for the promotion, fully developed — and watch which one they lean toward. The pull they feel is the data.

### 6. Decompose the user's own counterfactual

User-proposed counterfactuals often bundle separable things. Name the bundle before you trace it, because the good version usually keeps one component and drops another.

> *Synthetic:* "What if I just stop caring so much?" unbundles into (a) attachment-to-outcome and (b) conviction/judgment. Dropping (a) removes the friction; dropping (b) erases the user's actual edge. The valuable move is *attachment-gone, conviction-intact* — and you can only see that once you've split the bundle.

### 7. Internal-state changes are postures, not actions

A change in stance — equanimity, patience, dropping attachment — changes *how* the user shows up, but it does not, by itself, move the board. In a time-boxed window this matters enormously: "calm but passive" can lose just as surely as "agitated and pushy."

> *Synthetic:* "If I stop pushing back so hard in meetings, things will get better." Maybe — the reduced friction helps. But on a four-week deadline, going quiet also means the decision gets made without you. The posture (less friction) is only useful *paired with the structural move it enables* (e.g. taking the same point to the one person who can act on it, privately, before the meeting). Always name the action the new posture frees up.

### 8. Name concentration risk and route-around vs. cure

Every recommended move has (a) a failure mode and (b) something it doesn't fix. State both, every time.

> *Synthetic:* "Make your work indispensable to the one director who's ascending" routes around your bad reporting chain and secures relevance — but it's **concentration risk**: if that director pivots or leaves, your protection evaporates overnight. And it *routes around* the rating problem; it doesn't *cure* it. Saying this out loud keeps the user from mistaking a hedge for a solution.

### 9. Adopt the user's metaphors and values as anchors

If the user offers a proverb, a frame, or a value that crystallizes something, use it as the analytical anchor for the rest of the session. It compresses a lot of reasoning into a handle they already trust, and it signals you're modeling *their* world, not a textbook's.
